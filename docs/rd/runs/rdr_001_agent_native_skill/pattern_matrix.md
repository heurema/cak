# Pattern Matrix — RDR-001 Agent-Native Skill

Status: exploratory. Patterns are not architecture decisions.

Evidence status values:

- source-ledgered: supported by inspected `source_ledger.yaml` claims;
- seed: named input or local process pattern, not external evidence;
- lead: discovered but not inspected;
- speculative: useful hypothesis, not evidence.

| Pattern ID | Pattern | Source(s) | What to copy | What not to copy | Agent-native adaptation | Risk | Informs RDR questions | Evidence status |
|---|---|---|---|---|---|---|---|---|
| P01 | executable external skill library | `src_voyager_arxiv` | Keep reusable behavior outside model weights as interpretable artifacts. | Do not assume stable Minecraft-style APIs or feedback. | Require trace/replay, permissions, sandboxing, and verifier admission. | Unsafe or brittle executable actions. | Is a skill executable code? | source-ledgered |
| P02 | self-verification before skill admission | `src_voyager_arxiv` | Require execution-grounded evidence before accepting reusable behavior. | Do not treat self-verification as independent proof. | Pair self-verification with replay, external verifiers, and review. | Self-confirmation and sampled-test overfitting. | What evidence admits a skill? | source-ledgered |
| P03 | environment/execution feedback loop | `src_voyager_arxiv`, `src_skillrevise_arxiv` | Use execution errors, environment feedback, and re-execution to improve skills. | Do not generalize from one successful trace. | Preserve feedback as Evidence IR and gate promotion by replay/verifier results. | Trace-local repair can regress unseen cases. | What evidence does a skill carry? | source-ledgered |
| P04 | workflow induction from trajectories | `src_awm_arxiv` | Mine reusable routines from prior trajectories. | Do not assume retrieved workflows are valid in all similar tasks. | Attach preconditions, postconditions, and repair exits. | Wrong-state activation. | Is a skill a workflow? | source-ledgered |
| P05 | workflow brittleness in dynamic UI | `src_hmt_arxiv` | Treat flat trajectory memory as suspect across unseen sites. | Do not reuse concrete UI actions as universal procedure. | Decouple intent, stage, and action grounding. | Workflow mismatch. | Does StageGraph beat flat memory? | source-ledgered |
| P06 | abstract subroutine over concrete example | `src_awm_arxiv`, `src_hmt_arxiv` | Abstract repeated routines into reusable stages/subgoals. | Do not erase action-grounding constraints. | Compile abstract stage plus observable predicates into runtime candidates. | Overabstract workflows may lose actionable detail. | What is reusable procedural memory? | source-ledgered |
| P07 | Program Function should_activate / intervene | `src_hasp_arxiv` | Separate trigger predicate from intervention body. | Do not make every skill a hard override. | Represent active skills as state-conditioned Runtime IR. | False positives and overblocking. | Are PFs the active skill primitive? | source-ledgered |
| P08 | action override | `src_hasp_arxiv` | Study hard next-action modification. | Do not allow overrides without permission and rollback. | Gate override by policy, verifier, and risk level. | Hidden authority widening. | What powers can skills have? | source-ledgered |
| P09 | context injection | `src_hasp_arxiv` | Study softer corrective context injection. | Do not assume injected advice will be followed correctly. | Record injected context and outcome evidence. | Advice can be ignored or over-applied. | Are passive and active skills one model? | source-ledgered |
| P10 | structured intervention signals | `src_hasp_arxiv` | Preserve activation state, intervention mode, and outcome. | Do not leave interventions as opaque prompt edits. | Emit Runtime IR events for activation/intervention/effect. | Poor telemetry hides regressions. | What Runtime IR is minimal? | source-ledgered |
| P11 | strict filtering / library pollution prevention | `src_skillreducer_arxiv`, `src_hasp_arxiv`, `src_psn_arxiv` | Remove weak routing, non-actionable content, and unstable updates. | Do not optimize for token count or volume alone. | Use quality, utility, maturity, and security as admission signals. | Useful caveats can be compressed away. | How should skill libraries avoid decay? | source-ledgered |
| P12 | semantic skill contracts | `src_vaso_arxiv` | Give skills verifier-facing obligations. | Do not assume all agent state can be formalized. | Treat ContractSpec as a verifier facet, not the whole skill. | Verification depends on grounding correctness. | Is a skill a contract? | source-ledgered |
| P13 | model-checker filtering | `src_vaso_arxiv` | Filter inconsistent contracts where formal state exists. | Do not make model checking universal. | Tier verifiers: static, replay, formal when possible. | Overblocks useful informal skills. | How strong must verifier support be? | source-ledgered |
| P14 | counterexample-driven contract revision | `src_vaso_arxiv` | Convert verifier counterexamples into artifact-improvement evidence. | Do not auto-promote a revision from one counterexample. | Quarantine revisions until replay/security gates pass. | Overfitting to narrow counterexamples. | How do skills evolve safely? | source-ledgered |
| P15 | executable symbolic skill network | `src_psn_arxiv` | Model executable skills as compositional graph nodes. | Do not assume symbolic programs capture all LLM-agent state. | Use dependency/conflict edges only when behavior needs them. | Graph complexity. | Is SkillGraph necessary? | source-ledgered |
| P16 | maturity-aware update gating | `src_psn_arxiv` | Stabilize reliable skills while keeping uncertain ones plastic. | Do not make mature skills impossible to revise. | Add lifecycle states: draft, candidate, stable, protected, deprecated, quarantined. | Stale protected skills. | What lifecycle does a skill need? | source-ledgered |
| P17 | rollback validation | `src_psn_arxiv` | Validate updates/refactors against rollback criteria. | Do not treat rollback as only a Git operation. | Store rollback plans and evidence deltas for admitted skills. | External side effects may not roll back. | How should skill evolution be governed? | source-ledgered |
| P18 | stage-aware procedural memory | `src_hmt_arxiv` | Separate intent, stage, and action levels. | Do not retrieve flat traces as complete workflows. | Compile StageGraph-like memory into activation predicates plus action grounding. | Domain-specific state observability. | Is StageGraph needed? | source-ledgered |
| P19 | precondition/postcondition stage alignment | `src_hmt_arxiv`, `src_vaso_arxiv` | Validate reusable stages against observable pre/postconditions. | Do not trust natural-language predicates as checked truth. | Map predicates to typed state observations where possible. | Preconditions can be incomplete. | Can PFs, contracts, and stages share activation? | source-ledgered |

| P21 | progressive disclosure | `src_agent_skills_spec`, `src_skilljuror_arxiv`, `src_skillreducer_arxiv` | Load metadata first, instructions on trigger, resources/code as needed. | Do not rely on retrieval alone for correctness. | Pair disclosure with activation state, verifier gates, and telemetry. | Wrong resource loaded at wrong time. | What is portable skill interop? | source-ledgered |
