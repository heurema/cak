# Hypothesis Matrix — RDR-001 Agent-Native Skill

Status: exploratory. No hypothesis is adopted.

| Hypothesis | Description | Best supporting patterns | Weaknesses | Required Runtime IR | Required Evidence IR | Required verifier support | Security/admission implications | Failure modes | Minimal experiment | Kill criteria |
|---|---|---|---|---|---|---|---|---|---|---|
| H1: skill = portable package | A skill is a directory/package such as `SKILL.md` plus metadata, scripts, resources, and docs. Portable packages are good distribution shells but may not control runtime behavior. | Portable SKILL.md package; progressive disclosure; skill organization evaluation. | Package import is not activation, admission, authorization, or verification. Package fields can become a vague bundle. | Package metadata loader; resource loader; optional script execution envelope. | Author/source provenance; package manifest; resource list; test evidence; security scan record. | Schema/package validation; static checks; optional tests. | Requires trust tier, source audit, resource/code review, and quarantine for untrusted packages. | Agent loads the right package but applies it in the wrong state; malicious helper scripts; stale package resources. | Import the same package under no admission, static-only, verifier-gated, and security+provenance gates. | Kill as universal definition if package form cannot predict or constrain runtime behavior on identical traces. |
| H2: skill = executable code/program | A skill is reusable executable behavior, often code or a symbolic program, invoked by the agent. | Executable external skill library; self-verification; executable symbolic skill network. | Code does not define when it should run, what it may affect, or why it is trusted. Stable APIs may be assumed. | Execution sandbox; call interface; permissions/effects; dependency handling; trace capture. | Unit tests; execution traces; environment feedback; version history; dependency/provenance records. | Test execution; replay; static effect checks; sandbox policy checks. | High: code and dependencies need sandboxing, permission scopes, provenance, and resource-level vetting. | Unsafe side effects; wrong-state calls; hidden payloads; API drift. | Encode one repair as script/code and compare against text advice, PF, ContractSpec, StageGraph, and hybrid artifact. | Kill as universal definition if useful skills require non-code state, contracts, or governance not expressible in code alone. |
| H3: skill = runtime intervention / Program Function | A skill is an active runtime-control object with an activation predicate and an intervention such as context injection or action override. PFs are powerful but may overblock or require careful trigger precision. | Program Function should_activate/intervene; action override; context injection; structured intervention signals. | Active intervention can hide authority widening and block correct behavior if triggers are imprecise. | State snapshot; activation predicate; intervention mode; allowed effects; audit event; stop/rollback condition. | Failure traces; prevented-failure evidence; false-positive records; intervention outcome logs. | Shadow-mode activation evaluation; activation precision/recall; replay and overblocking metrics. | Requires approval boundaries, authority caps, auditability, and quarantine before active use. | False positives; oscillating interventions; action override bypass; ignored context injection. | Run PFs in shadow mode on recorded traces and compare predicted interventions to actual failures. | Kill if activation precision is too low or false positives block correct behavior above a defined threshold. |
| H4: skill = proof-carrying contract | A skill is a semantic contract or typed transition with verifier-facing obligations and counterexample-driven repair. Contract verification depends on grounding correctness. | Semantic skill contracts; model-checker filtering; counterexample-driven contract revision; pre/postcondition alignment. | Many agent environments lack reliable propositions, typed state, or complete observability. | Typed state; transition/effect model; contract hook; authority upgrade/downgrade path. | Grounding assumptions; obligations; counterexamples; verifier results; repair history. | Static checks; model checking where possible; replayed pre/postcondition checks; counterexample handling. | Admission depends on grounding audit and verifier scope; unsafe if proofs over wrong state are trusted. | False proof; over-formalization; useful skill blocked; verifier blind spot. | Express the same workflow as ContractSpec + repair handler, PF, StageGraph, and package. | Kill as universal definition if most useful skills cannot expose meaningful verifier-facing state. |
| H5: skill = stage-aware procedural memory | A skill is a reusable memory object with intent/stage/action hierarchy and observable preconditions/postconditions. Stage memory may be domain-specific and requires observable pre/postconditions. | Workflow induction; workflow brittleness; stage-aware procedural memory; precondition/postcondition stage alignment. | Stronger for web workflows than arbitrary tools; action grounding may remain domain-specific. | Stage graph; current-state observations; precondition checker; action-grounding selector; postcondition checker. | Source trajectories; abstraction notes; stage examples; mismatch/failure traces. | Pre/postcondition replay; state-observability checks; action-grounding validation. | Lower than code/PF when passive, but risky if it drives actions without admission. | Workflow mismatch; unobservable preconditions; stale UI/action selectors. | Encode the same workflow as linear workflow, state machine, StageGraph, PF, and ContractSpec + repair. | Kill as general definition if stage predicates cannot be observed or do not reduce mismatch versus simpler retrieval. |
| H6: skill = governed lifecycle node in SkillGraph | A skill is a graph node with dependencies, maturity, health, provenance, conflicts, rollback, and retirement. Lifecycle graph adds complexity and may be overengineering if skill count remains small. | Executable symbolic skill network; maturity-aware update gating; rollback validation; provenance graph. | Governance machinery can outrun validated need and obscure the active runtime primitive. | Registry node; dependency/conflict edges; lifecycle state; health metrics; rollback pointer. | Origin evidence; contribution scores; update history; diagnosis records; retirement rationale. | Utility tracking; conflict tests; rollback validation; pollution metrics. | Requires quarantine, maturity gates, conflict review, and provenance checks. | Bureaucracy; stale stable nodes; graph conflicts; false health metrics. | Simulate broad/narrow/conflicting skill candidates and compare flat registry vs lifecycle gates. | Kill if lifecycle metadata does not reduce pollution, conflict, or review load versus simple allowlist/quarantine. |
| H7: skill = compiled bridge artifact | A skill is an admitted compiled artifact bridging Evidence/Provenance IR to Runtime/Control IR. It may contain a Runtime IR fragment, Evidence IR links, verifier obligations, security envelope, and lifecycle state. | Portable package; PF intervention; semantic contracts; stage-aware memory; provenance graph; security envelope; diagnosis object. | May be overengineering; authoring burden and minimal viable subset are unknown. | Runtime IR fragment: activation predicate, intervention/workflow/contract hooks, allowed effects, permissions, lifecycle state. | Evidence IR links: source ledger claims, traces, diagnoses, counterexamples, provenance, version history, security scans. | Verifier obligations: schema/static checks, replay/shadow checks, contract checks where possible, utility thresholds. | Security envelope: source trust, resource/code audit, sandbox, context-aware authorization, quarantine, rollback. | Too many fields; unclear compiler boundary; no minimal experiment; package+tests may be enough. | Compile the same source package into candidate runtime artifacts under multiple admission gates and compare behavior/evidence cost. | Kill if package+tests or one simpler primitive matches behavior, safety, and auditability at much lower complexity. |
